# Prompt策略

## 目标

本项目优先利用当前 agent 自带的模型能力完成论文生成，规则系统主要承担三类职责：

- 限定模型可用的证据范围
- 控制单次生成的篇幅与批次
- 在输出前做结构和引用校验

## 核心原则

- 模型负责组织语言、构造段落、维持学术表达。
- 规则负责限定“只能用哪些资料、一次写多少、当前在写哪一章”。
- 不要求模型一次性生成整篇论文，必须拆成多章多批次完成。
- 默认由进入当前目录的 agent 直接读取 prompt 并执行，外部 API 调用只是可选扩展。

## 单章分批策略

推荐流程：

1. 先由 planning 生成大纲与章节目标。
2. writing 根据每章目标和证据块生成 prompt 包。
3. agent 读取对应 prompt 包，每章按 1 到多个批次生成正文。
4. 一个批次完成后，再继续下一个批次。
5. 全章完成后，再进入下一章。

这样做的主要原因：

- 避免单次输出 token 超限
- 降低模型在长输出中跑偏的概率
- 更方便在中途暂停、补资料、重写局部章节

## Prompt 结构

每个写作 prompt 至少包含：

- 当前章节名称
- 当前批次编号和建议字数
- 章节目标
- 可用证据块
- 硬约束
- 输出格式要求

## 硬约束模板

- 只能使用给定证据块，不得编造新事实。
- 不要跨章节写作。
- 不要提前写整篇结论。
- 不要一次性扩写成本章全部内容。
- 引用先使用临时源标记，例如 `[SRC-001]`。

## agent 交互要求

agent 在进入 writing 阶段后，应明确告诉用户：

- 当前写到第几章
- 当前是这一章的第几批
- 为什么要分批写
- 本批完成后下一步会做什么

建议说法：

```text
当前正在写第 3 章的第 1 批内容。
这一步不会一次性生成整章，而是先根据 3-4 个证据块写约 300-500 字，避免单次输出过长导致模型截断。
本批完成后，我会继续第 3 章的下一批，直到本章完成。
```

## 适用边界

- 适用于课程论文、综述型论文、现状分析、对策建议。
- 不适用于需要大规模原创推导或复杂实验结果复现的场景。

## 可选自动调用

如果某个 agent 运行环境额外配置了外部模型 API，可以使用 `src/writing/generate.py` 这类脚本批量回填章节草稿；但这不是本项目的默认依赖。
