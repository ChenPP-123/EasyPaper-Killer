# EasyPaper Killer 数据结构

## 1. 设计目标

数据结构的核心目标是让论文写作过程可追溯、可校验、可分阶段推进，避免生成内容和引用关系失控。

## 2. 核心对象

### 2.1 SourceRecord

表示一篇正式入库文献。

字段建议：

```json
{
  "source_id": "SRC-001",
  "title": "示例论文标题",
  "authors": ["作者A", "作者B"],
  "year": "2026",
  "journal": "示例期刊",
  "source_type": "journal",
  "cnki_url": "https://example.com",
  "abstract": "文献摘要",
  "citation_text_raw": "查新引文原文",
  "reference_text_raw": "参考文献原文",
  "file_paths": ["input/references/raw/pdf/示例.pdf"],
  "status": "formal"
}
```

说明：

- `status=formal` 表示可用于正式引用。
- 候选文献不应直接写入正式文献库，除非用户补齐资料。

### 2.2 EvidenceChunk

表示从文献中抽取出的可用证据片段。

字段建议：

```json
{
  "chunk_id": "EVD-001",
  "source_id": "SRC-001",
  "text": "该文献中可支持论文写作的关键内容",
  "location": "abstract",
  "evidence_type": "abstract_summary",
  "confidence": "high"
}
```

说明：

- `location` 可以是摘要、页码、章节名等。
- `confidence` 用于标记证据可靠程度。

### 2.3 OutlineNode

表示论文大纲中的一个章节节点。

字段建议：

```json
{
  "section_id": "SEC-01",
  "title": "研究背景与问题提出",
  "goal": "说明论文研究背景和问题意识",
  "allowed_source_ids": ["SRC-001", "SRC-002"],
  "target_words": 800,
  "notes": "尽量避免空泛政策口号"
}
```

说明：

- `allowed_source_ids` 很关键，用来限制每章允许使用哪些资料。

### 2.4 ClaimUnit

表示正文中的一个论断单元。

字段建议：

```json
{
  "claim_id": "CLM-001",
  "section_id": "SEC-01",
  "text": "江西省节能工作在十四五期间取得了阶段性成效。",
  "supporting_source_ids": ["SRC-001"],
  "supporting_chunk_ids": ["EVD-001"],
  "status": "supported"
}
```

说明：

- `status` 可以取值 `supported`、`weak`、`unsupported`。
- 只有 `supported` 状态的内容才适合直接进入终稿。

### 2.5 TemplateSpec

表示从模板中抽取的排版要求。

字段建议：

```json
{
  "template_name": "南信大毕业论文规范",
  "required_sections": ["题目", "摘要", "关键词", "正文", "参考文献"],
  "heading_rules": {
    "level_1": "黑体三号",
    "level_2": "黑体四号"
  },
  "body_font_rule": "宋体小四",
  "line_spacing_rule": "固定值或1.5倍行距",
  "reference_style_rule": "GB/T 7714-2015"
}
```

## 3. 文件级数据产物

### 3.1 `sources.json`

用途：保存正式文献元数据总表。

要求：

- 一条记录对应一篇正式入库文献。
- 必须能对应到原始资料文件。

### 3.2 `evidence.json`

用途：保存证据片段。

要求：

- 每条证据必须绑定 `source_id`。
- 禁止出现来源不明的证据片段。

### 3.3 `references.json`

用途：保存参考文献格式化结果。

要求：

- 要和正文引用编号可映射。
- 要能输出最终参考文献列表。

## 4. 状态设计

### 4.1 文献状态

- `formal`：用户已提供完整资料，可正式引用。
- `candidate`：仅为候选，不可直接正式引用。
- `discarded`：确认不纳入。

### 4.2 写作状态

- `draft`：草稿状态。
- `reviewing`：正在校验。
- `ready`：可进入导出阶段。

## 5. 数据约束

- 没有 `source_id` 的内容不能视为正式证据。
- 没有 `supporting_source_ids` 的论断不能直接进入终稿。
- 每个章节都应有明确的 `allowed_source_ids`。
- 候选文献必须和正式文献分开存储。

## 6. 面向 agent 的使用要求

agent 在对话中应隐式遵守这些结构，不需要让用户直接操作 JSON，但内部处理必须以这些对象为基础，才能保证后续写作和校验稳定。
